# ADR-017: CURAノート取込・基本シフト反映自動化

**ステータス**: 承認済み
**決定日**: 2026-03-19
**関連 Issue**: #280, #281, #282, #283, #284

---

## コンテキスト

現行の基本シフト作成フローでは、CURAポータルから自動振り分けされた「ノート」（スプレッドシート）の内容を、サ責が手作業でvisitcare-shift-optimizerのFirestoreオーダーに反映している。これが最大のボトルネック。

ノートの`スケジュール変更内容`列は**自由テキスト**であり、利用者名・日時・変更内容が非構造的に記載されている。ただし、一定のパターン（`{利用者名}様(...)` で始まり、内容にキャンセル/時間変更/受診同行等のキーワードを含む）が存在する。

---

## 検討した選択肢

### A. 自由テキストをLLMで解析

| 項目 | 評価 |
|------|------|
| 精度 | 高（文脈理解可能） |
| コスト | API呼び出し毎に課金 |
| レイテンシ | 2-5秒/件 |
| 複雑度 | プロンプト管理、ハルシネーション対策 |

### B. 正規表現 + キーワードマッチング（採用）

| 項目 | 評価 |
|------|------|
| 精度 | 中（パターン外は手動分類） |
| コスト | ゼロ |
| レイテンシ | ミリ秒 |
| 複雑度 | パターン定義・保守 |
| 透明性 | 高（ルールが明示的） |

### C. LLMとルールのハイブリッド

| 項目 | 評価 |
|------|------|
| 精度 | 高 |
| コスト | ルール外のみAPI課金 |
| 複雑度 | 高（2系統の保守） |

---

## 決定

**選択肢B: 正規表現 + キーワードマッチング** を採用。

### 理由

1. **ノートの自由テキストには一定のパターンがある** — 利用者名は必ず先頭行に `{name}様` 形式、アクション種別はキーワード（キャンセル、時間変更、受診等）で判定可能
2. **透明性** — ルールが明示的で、なぜその判定になったかサ責が理解・修正できる
3. **コスト** — 追加のAPI課金なし
4. **段階的改善** — パターン外のケースを収集し、将来的にLLM解析（選択肢C）へ段階移行可能
5. **解析不能なケースは`unknown`として人間に委ねる** — 無理に自動化しない

## アーキテクチャ

```
CURAノート（Google Sheets）
    ↓ Google Sheets API（読み取り）
sheets_reader.py — 未処理行（対応可否=0）を取得
    ↓ NoteRow（Pydantic）
note_parser.py — 自由テキスト解析 → ParsedNote
    ↓
note_diff.py — Firestoreオーダーとマッチング → NoteImportAction[]
    ↓
API: POST /import/notes → NoteImportPreview（dry-run）
    ↓ ユーザー確認
API: POST /import/notes/apply → Firestore反映 + シート更新
```

### 解析フロー

1. `スケジュール変更内容` から利用者名を抽出（`{name}様` パターン）
2. `入退院・その他` と テキスト内キーワードからアクション種別を判定:
   - `キャンセル` → delete（order.status = cancelled）
   - `時間変更` / `→` → update（start_time/end_time変更）
   - `受診同行` / `受診` → add（臨時オーダー追加）
   - `ヘルパーの休み` → staff_unavailability登録
   - 判定不能 → unknown（手動対応）
3. `日付From/To` で対象日を特定
4. Firestoreの既存オーダーとマッチング（customer + date + time帯）

### APIエンドポイント

| エンドポイント | メソッド | 概要 |
|--------------|--------|------|
| `/import/notes` | POST | ノート読み取り → 差分プレビュー |
| `/import/notes/apply` | POST | プレビュー内容をFirestoreに反映 |

---

## 影響

- `optimizer/src/optimizer/integrations/` ディレクトリ新規作成
- `routes.py` にエンドポイント2つ追加
- フロントエンドにインポートUI追加
- 既存コードへの破壊的変更なし

---

## リスク

| リスク | 対策 |
|-------|------|
| テキスト解析の精度 | unknownケースを人間に委ね、パターンを段階的に追加 |
| 利用者名の表記揺れ | カナ・漢字・短縮名での fuzzy matching |
| 同日同利用者の複数オーダー | 時間帯でのマッチング精度向上 |
| シートの列構成変更 | 列名ベースのマッピング（インデックス非依存） |
