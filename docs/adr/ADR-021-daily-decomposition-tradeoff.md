# ADR-021: 日次分割による週次最適化品質のトレードオフ

## ステータス
承認済み（2026-03-31）

## コンテキスト

PR#333で導入した`decompose_by_day=True`（デフォルト）により、最適化問題を日付ごとに分割して独立にソルブするようになった。これにより本番規模（10,050オーダー/176ヘルパー）が2GB以内・5分以内で処理可能になったが、週次ソフト目的関数が日単位に分断される副作用がある。

### 影響を受けるソフト目的

| 目的 | 週次一括時の挙動 | 日次分割時の挙動 | 品質影響 |
|------|----------------|----------------|---------|
| ワークロードバランス | 週合計で preferred_hours の超過/不足にペナルティ | 日ごとに独立評価 | 中：日次でも1日の過負荷は検出可能 |
| 担当継続性 | 週を通じて同一利用者のスタッフ数を最小化 | 日ごとに独立評価（4件未満/日の利用者はスキップ） | 高：週5日×1件/日の利用者は継続性ペナルティが効かない |

## 決定

**日次分割の品質劣化を仕様として受容する。**

### 理由

1. **スケーラビリティが最優先**: OOM（28GB）を解消しないと本番デプロイ不可
2. **継続性の実質的影響は限定的**: 本番データでは1日4件以上の大口利用者にのみ継続性が有効に作用する。小口利用者の継続性はサ責の手動調整で補完可能
3. **フォールバック可能**: `decompose_by_day=False` をAPIパラメータで指定すれば週次一括に戻せる（小規模データ向け）
4. **Phase 2で改善予定**: 週間リバランスパス（日次分割後に週次ソフト目的で微調整する2パス方式）を検討

## 代替案

1. **週次一括のまま変数削減のみ**: メモリは改善するが10,050オーダーでは依然として制限時間超過のリスク → 不採用
2. **2パス方式（日次分割 + 週次リバランス）**: 理想的だが実装コストが高く、Phase 2で対応 → 将来検討
3. **ヒューリスティック前処理で問題サイズ削減**: 効果が予測しにくく検証コストが高い → 不採用

## 影響

- `solver.py` の `solve()` 関数に NOTE/TODO コメント追加
- `test_decomposition.py` に日次分割時の継続性テスト追加（1日4件以上のケース）
- UIへの影響なし（ソフト目的のスコアはユーザーに直接表示されない）
