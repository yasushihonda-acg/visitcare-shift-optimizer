# ADR-018: AIエージェント方式への戦略的ピボット

**ステータス**: 提案中
**決定日**: 2026-03-23
**関連 ADR**: ADR-001, ADR-006, ADR-017

---

## コンテキスト

Phase 6d完了時点で、visitcare-shift-optimizerは数理最適化エンジン（PuLP + CBC）を中心としたシフト自動生成システムとして稼働している。CURAノート取込（ADR-017）、当週シフト一括複製、休み希望反映、翌日チェックリスト等の自動化機能を実装済み。

クライアントとの打合せ（2026-03-23）で以下の課題・ニーズが明らかになった:

1. **ブラックボックス問題**: 数理最適化の結果は「なぜこのシフトになったか」が見えにくく、サ責が結果を信頼・採用しにくい
2. **段階的・対話的な業務実態**: 実務ではサ責がヘルパーの状況・利用者の要望を踏まえながら段階的にシフトを組んでおり、一括最適化より対話的なアプローチが自然
3. **ヘルパー個別支援ニーズ**: シフト管理だけでなく、各ヘルパーのメンタルケアや業務お困りごと対応へのニーズがある
4. **コントローラビリティ**: 人間が判断に関与しながらAIの支援を受ける方が現場に受け入れられやすい

- AsIsシステム構成図: `docs/diagrams/asis-system-architecture.drawio.xml`
- AsIs業務フロー図: `docs/diagrams/asis-business-flow.drawio.xml`

## 検討した選択肢

### A. 現行の数理最適化路線を継続

| 項目 | 評価 |
|------|------|
| 制約遵守の確実性 | 高（数理的保証） |
| ユーザー受容性 | 低（ブラックボックス） |
| 拡張性（メンタルサポート等） | 低 |
| 実装コスト | 低（既存資産活用） |

### B. AIエージェント＋チャットボット形式へピボット（採用）

| 項目 | 評価 |
|------|------|
| ユーザー受容性 | 高（対話的、段階的） |
| 拡張性 | 高（ヘルパー支援、RAGコンテキスト） |
| 制約遵守の確実性 | 中（LLMの確率的挙動） |
| 実装複雑度 | 高（LLM/RAG/チャットUI） |

### C. ハイブリッド（数理最適化 + AIエージェント）

| 項目 | 評価 |
|------|------|
| 制約遵守 | 高（数理側で担保） |
| ユーザー体験 | 高（AI対話 + 最適化） |
| 複雑度 | 最高（2系統の統合） |

## 決定

**選択肢B: AIエージェント＋チャットボット形式へピボット**する。

2系統のAIエージェントを構築する:

### 1. シフト管理AIエージェント

- **対象ユーザー**: サ責（サービス提供責任者）
- **機能**: CURAデータ・ヘルパー情報・利用者情報をRAGコンテキストとして保持し、サ責との対話を通じてシフトを段階的に生成
- **業務フロー**: 基本シフト→当週→当日の4フェーズを対話的にサポート
- **インターフェース**: チャットボット形式（Webアプリ内）

### 2. ヘルパー支援AIエージェント

- **対象ユーザー**: 各ヘルパー（ヘルパーごとに個別エージェント）
- **機能**: メンタルサポート＋業務お困りごと対応
- **コンテキスト**: 担当利用者情報、過去の業務記録、スケジュール等をRAG的に保持
- **個別化**: ヘルパーごとにパーソナライズされたコンテキスト

### 既存最適化エンジン（PuLP+CBC）の位置づけ

**現時点では未決定**。以下の選択肢を今後検討する:

- (a) AIエージェントの内部ツールとして併用（制約チェック・候補生成用）
- (b) 段階的に廃止（AIエージェントが完全に代替）
- (c) バックアップ・検証用として維持

→ 次回ADRで決定予定。既存optimizer/ディレクトリのコードは当面維持する。

参照ダイアグラム: `docs/diagrams/tobe-ai-agent-architecture.drawio.xml`

## 理由

1. **業務フローとの親和性**: 現場のサ責は対話的・段階的にシフトを組む業務フローが定着しており、一括最適化より対話型が自然
2. **ユーザー受容性**: 「AIに相談しながら決める」方が「AIが出した結果を採用する」より心理的ハードルが低い
3. **価値拡張**: ヘルパー個別のメンタルサポート・業務支援という新たな価値提供が可能
4. **データ資産活用**: CURAデータ・Firestoreのヘルパー/利用者/オーダーデータをRAGで活用できる

## 影響

### 新規技術要素

| 要素 | 内容 | 備考 |
|------|------|------|
| LLMプロバイダ | Gemini / Claude / OpenAI等 | 未選定。次フェーズで検証・ADR化 |
| RAGパイプライン | Firestoreデータ + CURAデータの検索・取得 | Embedding + Vector Search or 構造化検索 |
| チャットUI | Webアプリ内チャットインターフェース | 既存Next.jsアプリに追加 |
| セッション管理 | エージェントごとの会話履歴・コンテキスト | ヘルパー数 × セッション |

### 既存資産の活用

- **Cloud Firestore**: 顧客/ヘルパー/オーダーデータはRAGコンテキストの主要データソースとしてそのまま活用
- **Firebase Auth (RBAC)**: サ責/ヘルパーのロール分離はエージェントアクセス制御に流用
- **CURA連携（Sheets API）**: ノート/ふせん/チェックリスト取込はRAGデータソースとして継続活用
- **既存UI（ガントチャート等）**: シフト表示・手動編集機能はAIエージェントの出力ビューとして併用

### ADR影響

| ADR | 影響 |
|-----|------|
| ADR-001 (数理最適化採用) | 主要アプローチとしての位置づけが変化。ステータス変更は次フェーズで判断 |
| ADR-006 (PuLP+CBC) | 同上。エンジン自体は当面維持 |
| ADR-017 (CURAノート取込) | RAGデータソースとして継続活用。影響なし |

## リスク

| リスク | 影響度 | 対策 |
|-------|--------|------|
| LLMのハルシネーション（不正確なシフト提案） | 高 | 制約チェックレイヤーの導入、人間の確認ステップ必須 |
| RAGコンテキストの鮮度 | 中 | Firestoreリアルタイムリスナーとの統合 |
| LLM APIコスト | 中 | トークン使用量モニタリング、キャッシュ戦略、モデル選定 |
| ヘルパー個別エージェントのスケーラビリティ | 中 | セッション管理設計、コスト見積もり |
| 既存最適化エンジンとの統合方針未決定 | 低 | 次フェーズで検証・ADR化 |
